# v1.3: Execution Workflow

## Objective

Turn published routes into operational route runs with status tracking, exception handling, dispatcher intervention, and proof capture.

## Core Workflow

- Route run status lifecycle
- Stop status lifecycle
- Dispatcher live board
- Exception creation
- Exception resolution
- Proof of service or delivery
- Dispatcher notes
- Event timeline

## Route Run Statuses

- `not_started`
- `dispatched`
- `en_route`
- `in_progress`
- `completed`
- `failed`
- `canceled`

## Stop Statuses

- `not_started`
- `en_route`
- `arrived`
- `serviced`
- `failed`
- `canceled`
- `rescheduled`
- `closed`

## Must Support

- Dispatch a route run
- Start and complete a route run
- Mark stops en route, arrived, serviced, failed, or rescheduled
- Add dispatcher notes
- Add proof metadata
- Open and resolve exceptions
- Inspect route and stop timelines

## Read Models

Build these query surfaces:

- `GET /dispatch/board?serviceDate=YYYY-MM-DD`
- `GET /route-runs/:id`
- `GET /route-runs/:id/timeline`
- `GET /route-run-stops/:id/timeline`
- `GET /exceptions`

## Exit Criteria

- Dispatcher can manage the full day-of-operations flow from a live board
- Each route run has durable lifecycle state
- Each route stop has actual timestamps and event history
- Exceptions can be acknowledged and resolved
- Proof artifacts and notes are attached to route run stops
