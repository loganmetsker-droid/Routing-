# v1.0: Stabilize Direction

## Objective

Turn the current routing product into a narrow dispatch planning and execution SaaS foundation for `local delivery`.

## Decisions Locked In This Version

- Initial operating model: `local delivery`
- Planning horizon: `single-day`
- Depot model: `single primary depot`
- Primary user: `dispatcher`
- Backend shape: `modular monolith`
- Optimizer shape: `internal module`
- Product focus: `planner + execution + exception handling`

## Build Now

- Separate planning from execution as a product rule and a data rule
- Make `jobs`, `job_stops`, `route_plans`, `route_runs`, and `assignments` first-class
- Treat manual dispatcher edits as a core feature, not a fallback
- Add auditability before expansion into broader SaaS features
- Add organization boundaries early enough to avoid global tables later

## Do Not Build

- Generalized multi-industry routing
- Enterprise fleet suite features
- Driver-native app before the dispatcher workflow is stable
- Optimizer platform or optimization API product
- Research-grade solver work

## Repo Direction

Use the existing repo as a modular monolith and converge backend modules toward:

- `organizations`
- `users`
- `depots`
- `drivers`
- `vehicles`
- `jobs`
- `planning`
- `route-plans`
- `route-runs`
- `assignments`
- `dispatch`
- `exceptions`
- `proofs`
- `audit`
- `realtime`

## Immediate Refactors

- Remove assumptions that `route = execution`
- Stop returning optimizer output as UI-shaped route blobs
- Give every stop stable identity instead of array-position-only modeling
- Require assignment and publish state before a route is operationally valid

## Exit Criteria

- Product scope is explicitly narrowed to local delivery dispatch
- Planning and execution are treated as separate flows
- Module and schema refactor targets are documented and accepted
- Future work is blocked from re-expanding into broad fleet scope
