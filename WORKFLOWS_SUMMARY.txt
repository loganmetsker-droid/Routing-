  HOW TO USE:
═══════════════════════════════════════════════════════════════

BASIC COMMANDS:
  /review     - Run code quality review
  /security   - Run security vulnerability scan
  /design     - Run UI/UX and accessibility review

TYPICAL WORKFLOW:
  1. Write code
  2. Run /review before committing
  3. Fix any issues
  4. Run /security before PR
  5. If UI changes, run /design (requires running app)
  6. Create PR

      "args": ["-y", "@modelcontextprotocol/server-playwright"]

  }

  - Automated accessibility checks
  - Responsive design verification

═══════════════════════════════════════════════════════════════
  WHAT EACH WORKFLOW CHECKS:
═══════════════════════════════════════════════════════════════

CODE REVIEW (/review):
  ✓ Syntax errors and completeness
  ✓ Code style consistency
  ✓ Potential bugs and logic errors
  ✓ Best practices adherence
  ✓ TypeScript usage
  ✓ NestJS/React patterns

SECURITY REVIEW (/security):
  ✓ SQL/NoSQL injection
  ✓ XSS (Cross-site scripting)
  ✓ Command injection
  ✓ Authentication bypasses
  ✓ Authorization flaws
  ✓ Hardcoded secrets
  ✓ Weak cryptography
  ✓ Data exposure

DESIGN REVIEW (/design):
  ✓ Visual consistency
  ✓ Color palette adherence
  ✓ Typography scale
  ✓ WCAG AA+ accessibility
  ✓ Keyboard navigation
  ✓ Screen reader compatibility
  ✓ Responsive design
  ✓ Component patterns

═══════════════════════════════════════════════════════════════
  SEVERITY LEVELS:
═══════════════════════════════════════════════════════════════

CRITICAL (Security HIGH):
  → Block deployment
  → Fix immediately

HIGH:
  → Fix before merge
  → Required for PR approval

MEDIUM:
  → Fix in current sprint
  → Or document as tech debt

LOW:
  → Address during refactoring
  → Or when convenient
